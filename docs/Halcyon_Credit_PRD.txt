Regulatory  Validation 
Validate  applicant  information  against  mock  regulatory  services  (identity,  employment,  tax,  sanctions).  Retry  policy  and  human-escalation  fallback  must  be  implemented.  The  system  must  never  fabricate  a  regulatory  response. 
Verification  status  ·  Compliance  findings  ·  Escalation  trigger  if  unresolved 
FR-7 
Fairness  Audit Evaluate  every  decision  for  disparate  impact  across  protected  and  proxy  demographic  groups.  Raise  a  bias  alert  and  trigger  escalation  if  the  fairness  gap  exceeds  the  guardrail  threshold. 
Fairness  indicators  ·  Bias  alerts  ·  Segment-level  approval-rate  delta 
FR-8 
Recommendation  Generation 
Generate  one  of  three  recommendations  —  Approve,  Decline,  or  Refer  to  Human  Review  —  supported  by  risk  score,  policy  findings,  regulatory  checks,  document  findings,  and  fairness  result.  The  system  may  not  emit  Approve  or  Decline  without  a  complete  evidence  chain. 
Recommendation  ·  Evidence  chain  ·  Escalation  flag 
FR-9 
Explanation  Generation 
Produce  a  human-readable,  auditable  rationale  grounded  in  retrieved  policy  clauses,  applicant  data,  and  ML  model  outputs.  Every  factual  claim  in  the  explanation  must  be  traceable  to  a  source. 
Narrative  explanation  ·  Policy  references  ·  Supporting  applicant  data  ·  Risk  model  factors 
FR-10 
Audit  Logging Record  a  complete,  immutable  decision  record  for  every  application  processed. 
Inputs  ·  ML  outputs  ·  Policy  checks  ·  Regulatory  checks  ·  Document  findings  ·  Fairness  result  ·  Final  recommendation  ·  Timestamp 
FR-11 
Human-in-the-Loop  Workflow 
Route  applications  meeting  any  escalation  criterion  to  a  human  underwriter  queue.  Escalation  triggers:  missing  information,  ML  confidence  below  threshold,  policy  conflict,  regulatory  verification  failure,  fairness  alert,  thin-file  flag. 
Escalation  record  ·  Reason  code  ·  Underwriter  queue  entry

Latency the  time  until  the  system  returns  a  complete,  auditable  recommendation≤  10  s.  Maximum  (P95)  ≤  20  s.  ML  inference  budget  ≤  300  ms.  RAG  retrieval  budget  ≤  600  ms. 
Availability Target  uptime  ≥  99%  during  pilot  operations. 
Cost  ceiling Maximum  cost  per  assessed  application  <  $0.05.  Guardrail  alarm  triggers  human-path  fallback  if  exceeded.

Explainability Every  recommendation  must  contain  an  evidence  chain:  applicant  data  references,  ML  risk  factors  (SHAP),  retrieved  policy  clauses  (with  source  IDs),  regulatory  check  result,  fairness  result. 
Audit  trail Every  decision  record  is  immutable  and  queryable.  Compliance  officer  must  be  able  to  reconstruct  the  full  reasoning  path  from  stored  artefacts  alone. 
Human  fallback Any  failure  in  ML  inference,  RAG  retrieval,  document  verification,  or  regulatory  check  must  degrade  to  human-review  path.  The  system  must  never  auto-decide  on  a  failed  sub-component. 
Security API  keys  managed  via  secrets  manager,  never  in  code  or  prompts.  Least-privilege  tool  access.  Audit  logging  enabled  for  all  external  service  calls.

  # Assumption Status Comment 
A-1 Historical  repayment  outcomes  (ground-truth  labels)  are  available  in  the  chosen  dataset. 
Confirmed 
HC  2018  application_train.csv  contains  the  TARGET  column  fully  populated  across  all  307,511  rows  (0  =  repaid  on  time,  1  =  payment  difficulty).  Zero  missing  values.  Ground-truth  label  is  available  as-is  for  ML  training  and  held-out  evaluation. 
A-2 A  synthetic  lending  policy  rulebook  exists  and  can  be  used  as  clean  text  for  RAG  indexing. 
Confirmed 
A  synthetic  lending  policy  rulebook  of  15–25  clauses  will  be  authored  for  the  pilot,  covering  DTI  thresholds,  income  minimums,  loan-to-income  limits,  thin-file  escalation  rules,  documentation  requirements,  and  fairness  guardrails.  This  corpus  is  the  RAG  retrieval  target  for  the  de-risk  spike  and  Sprint  1. 
A-3 Human  underwriters  remain  the  final  decision  authority  —  no  autonomous  approve/decline  in  v1. 
Confirmed 
Explicitly  stated  in  the  capstone  brief  and  locked  in  PRD  Section  11  (Human-in-the-Loop  Policy).  The  system  generates  recommendations  only.  No  downstream  disbursement  or  loan  management  action  is  triggered  by  system  output. 
A-4 Home  Credit  Default  Risk  2018  is  representative  enough  of  the  thin-file  borrower  population  for  pilot  experimentation. 
Confirmed 
HC  2018  was  built  explicitly  for  financial  inclusion  and  underbanked  borrower  populations,  directly  matching  Halcyon  Credit's  stated  problem.  Confirmed  as  primary  dataset  for  pilot  scope. Caveat:  Geographic  and  temporal  distribution  may  differ  from  Halcyon's  real  applicant  pool.  Real-world  deployment  requires  retraining  on  proprietary  data.  This  caveat  will  be  documented  in  the  dataset  card. 
A-5 Bureau  data  is  available  as  structured  fields  —  no  unstructured  bureau  narrative  in  v1. Confirmed 
HC  2018  bureau.csv  and  bureau_balance.csv  are  entirely  structured  —  credit  type,  outstanding  amounts,  days-past-due  history,  status  codes  (C/X/0–5),  and  monthly  balance  history.  No  unstructured  narrative  text  exists  in  either  bureau  table.  Mock  regulatory  API  calls  in  v1  will  return  structured  JSON  responses  consistent  with  this  schema. 
A-6 Third-party  LLM  API  is  acceptable;  on-premise  inference  is  not  required  for  the  pilot. 
Confirmed 
Brief  permits  third-party  APIs  and  free  cloud  tiers  but  PII  masking  layer  is  required
